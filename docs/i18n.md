# 🌐 Diretrizes Globais de Internacionalização (i18n)

Este documento estabelece as diretrizes arquiteturais, o catálogo de termos e a estratégia de execução para a internacionalização total da plataforma SaaS (Next.js App Router).

---

## 🏗️ 1. Arquitetura i18n no Next.js App Router

Como a plataforma utiliza o **Next.js App Router (v16.1.6)**, temos três abordagens possíveis para gerenciar múltiplos idiomas (Português e Inglês). 

### Comparativo de Abordagens

| Abordagem | Prós | Contras | Recomendação |
| :--- | :--- | :--- | :--- |
| **Opção A: Sub-roteamento por URL (`/[locale]/...`) com `next-intl`** | Excelente para SEO nas páginas públicas; compatível com Server Components; padrão recomendado pelo Next.js. | Requer mover todas as páginas de `src/app/*` para `src/app/[locale]/*`. | **Altamente Recomendado para Páginas Públicas (`/ranking`, `/mentor`, `/v`)** |
| **Opção B: Tradução Baseada em Estado/Cookie (Flat URLs)** | Mantém URLs limpas (ex: `/admin` continua `/admin` em qualquer idioma); sem necessidade de reestruturar diretórios de arquivos. | Pior indexação SEO em rotas públicas caso dependa exclusivamente do lado do cliente. | **Recomendado para o Painel Administrativo (`/admin`)** |
| **Opção C: Arquitetura Híbrida (Recomendada)** | O melhor de dois mundos. Páginas públicas usam sub-roteamento para SEO, e o dashboard do admin usa flat URLs baseadas nas preferências do usuário/cookie. | Ligeiramente maior complexidade de configuração inicial de middleware. | **Solução Ideal para este Projeto** |

### Fluxo de Decisão de Idioma (Middleware)
1. **Preferência Explícita**: Cookie `NEXT_LOCALE` ou preferência salva no banco de dados (MongoDB para usuários autenticados).
2. **Cabeçalho de Idioma**: Detecção automática do idioma do navegador via cabeçalho `Accept-Language`.
3. **Fallback**: Padrão para Português (`pt`), com suporte completo a Inglês (`en`).

## 🚀 2. Diretrizes de Execução Técnica

### A. Tradução de Componentes React (Client & Server)
Para traduzir strings de interface nos arquivos `.tsx`, utilizaremos o hook `useTranslations` (se utilizarmos `next-intl`) ou um hook customizado conectado ao Zustand:

```tsx
// Exemplo prático de uso
import { useTranslations } from 'next-intl';

export function Header() {
  const t = useTranslations('common');
  return (
    <button className="bg-black text-white px-4 py-1.5 rounded-full">
      {t('signUp')}
    </button>
  );
}
```

### B. Tradução Dinâmica de Atributos e App Tags
Os metadados das App Tags (definidos em `src/lib/app-tags.ts`) devem conter dicionários traduzíveis em vez de strings estáticas:

```typescript
// Nova estrutura proposta em src/lib/app-tags.ts
export interface AppAttribute {
  id: string;
  labelKey: string;       // Chave de tradução ex: "attributes.product_category.label"
  placeholderKey: string; // Chave de tradução ex: "attributes.product_category.placeholder"
  appTagId: AppTagId;
}
```

### C. Tradução de Prompts de IA (Motor de Geração)
Os prompts enviados para o OpenAI (definidos em `src/lib/templates.ts`) devem ser adaptados dinamicamente para instruir a IA a responder no idioma ativo do usuário.
* Adicionar a variável `{locale_instruction}` no final de cada prompt básico.
* **Exemplo**:
  * Para `pt`: `"Escreva sua resposta estritamente em português."`
  * Para `en`: `"Write your response strictly in English."`

## 🛠️ 4. Cronograma de Implementação (Passo a Passo)

### Fase 1: Fundação & Infraestrutura (Sprint 1)
1. Instalar as dependências necessárias (`npm install next-intl` se for a opção híbrida, ou configurar o provider de estado).
2. Criar a pasta `messages/` na raiz do projeto com arquivos `pt.json` e `en.json`.
3. Configurar o arquivo `src/i18n.ts` de bootstrap de traduções.
4. Criar o middleware de i18n para interceptar rotas e detectar cookies/preferências.

### Fase 2: Adaptação das Páginas Públicas (Sprint 2)
1. Mover rotas públicas do App Router (`src/app/ranking`, `src/app/v`, `src/app/mentor`, `src/app/mentorado`, etc.) para a pasta `src/app/[locale]/`.
2. Substituir todas as strings estáticas em português nessas páginas por chamadas do hook `useTranslations()`.

### Fase 3: Adaptação da Home & Onboarding (Sprint 3)
1. Mover a rota raiz `/` para a estrutura de idiomas.
2. Traduzir o assistente virtual de chat em `HomeContainer.tsx` e `ChatInterface.tsx`.
3. Alterar `src/lib/app-tags.ts` para carregar rótulos do arquivo de idiomas.

### Fase 4: Adaptação do Admin & Módulos ADE (Sprint 4)
1. Traduzir as barras de navegação, painéis laterais e cabeçalhos em `/admin`.
2. Traduzir consecutivamente os grandes componentes de cada App Tag (`TradeRankingMeter.tsx`, `LogisticsBoard.tsx`, `StoreLayoutGrid.tsx`, `MentoringProfileBoard.tsx`, etc.).
3. Adicionar uma chave de idioma de geração no back-end para que as IAs criem conteúdos traduzidos coerentemente.

---

> [!IMPORTANT]
> A internacionalização da plataforma deve preservar a fidelidade visual dos componentes. Atenção especial deve ser dada a textos em inglês que costumam ser ligeiramente menores ou maiores do que em português, para evitar quebras em layouts premium com HSL, glassmorphism e micro-animações.
