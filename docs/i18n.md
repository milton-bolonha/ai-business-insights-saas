# 🌐 Diretrizes Globais de Internacionalização (i18n) e Localização (l10n)

Este documento estabelece as diretrizes arquiteturais, o catálogo de termos, as regras linguísticas e a estratégia de execução para a localização em nível de produção da plataforma SaaS (Next.js App Router).

---

## 🏗️ 1. Arquitetura i18n no Next.js App Router

A plataforma utiliza o **Next.js App Router (v16.1.6)**. Adotaremos uma **Arquitetura Híbrida** para otimização máxima de SEO e experiência do usuário:

*   **Páginas Públicas & Home (`/ranking`, `/mentor`, `/v`, `/`):** Sub-roteamento baseado em URLs com a biblioteca `next-intl` usando o prefixo `[locale]` (ex: `/pt/ranking` ou `/en/ranking`). Isso assegura a indexação perfeita por robôs de busca (SEO) em múltiplos idiomas.
*   **Painel Administrativo (`/admin`):** URLs limpas (flat URLs) onde o idioma é detectado pelas preferências do usuário salvas no MongoDB/Clerk e persistido em um cookie (`NEXT_LOCALE`), mantendo a rota administrativa limpa e fácil de compartilhar.

## 🚀 2. Diretrizes Técnicas de Desenvolvimento (Padrões Globais)

### A. String & Content Management (Sem Textos Hardcoded)

1.  **Externalização Total**: Nenhum texto legível pelo usuário deve estar hardcoded na lógica da aplicação. Todas as strings devem ser extraídas como pares de chave-valor em arquivos JSON de recursos (`messages/pt.json` e `messages/en.json`).
2.  **Sem Concatenação de Strings**: É proibido juntar strings (ex: `"Olá, " + nome` ou `"Você tem " + count + " itens"`). A ordem das palavras muda drasticamente entre idiomas. Use **placeholders e formatação baseada em ICU**:
    *   *Errado*: `<span>Olá, ` + `<b>{username}</b>` + `</span>`
    *   *Correto*: `t('welcomeMessage', { username })` com a tradução `"welcomeMessage": "Olá, {username}"`

### B. Linguística & Formatos ICU (Pluralização e Menus)

1.  **Pluralização Nativa com ICU**: Evite condicionais personalizados (`if (count === 1) ...`). Use o formato de mensagem ICU para gerenciar regras de pluralização complexas:
    *   *Exemplo no JSON*:
        ```json
        "activeWorkspaces": "{count, plural, =0 {Nenhum workspace ativo} =1 {1 workspace ativo} other {# workspaces ativos}}"
        ```
    *   *Uso no Componente*: `t('activeWorkspaces', { count: activeWorkspaces.length })`
2.  **Menus e Metadados Externalizados**: Menus de navegação, cabeçalhos, categorias e as App Tags (definidas em `src/lib/app-tags.ts` e `src/lib/templates.ts`) não devem possuir strings literais em TypeScript. Elas conterão chaves de tradução (`labelKey`, `placeholderKey`) que serão interpretadas em tempo de execução:
    *   *Exemplo*:
        ```typescript
        export const APP_TAGS = [
          {
            id: "love_writers",
            labelKey: "appTags.love_writers.label",
            color: "#e11d48",
          }
        ];
        ```

### C. UI Layout, Flexibilidade & RTL

1.  **Text Expansion & Contraction**: Traduções para idiomas como o alemão podem expandir o texto em 30% a 40%, enquanto idiomas asiáticos costumam contrair.
    *   **Regra**: Nunca use larguras fixas (`w-[120px]`) para botões ou elementos contendo texto. Use layouts flexíveis (`flex`), grids responsivos e `min-w-*` com margens dinâmicas.
2.  **Suporte a RTL (Right-to-Left)**: Para futuras expansões para idiomas como Árabe ou Hebraico, os estilos devem utilizar propriedades lógicas do CSS em vez de direcionais físicas:
    *   Use `padding-inline-start` (`ps-*` no Tailwind) em vez de `padding-left`.
    *   Use `margin-inline-end` (`me-*` no Tailwind) em vez de `margin-right`.
    *   Adicione a diretiva `dir={locale === 'ar' ? 'rtl' : 'ltr'}` na tag `<html>` no layout global.

### D. Formatação e Sensibilidade a Locales (Data, Hora e Números)

1.  **Números e Separadores**: Separadores de milhar e decimal mudam (ex: `1,234.56` nos EUA é `1.234,56` no Brasil).
    *   Use a API nativa do navegador `Intl.NumberFormat` ou utilitários do `next-intl` para formatação:
        ```typescript
        const formatted = new Intl.NumberFormat(locale, { style: 'currency', currency: 'BRL' }).format(valor);
        ```
2.  **Datas e Horários**: Formatos de exibição diferem (`DD/MM/YYYY` vs `MM/DD/YYYY`).
    *   Use `Intl.DateTimeFormat` para renderizar datas locais.
    *   **Regra de Banco de Dados**: Todos os timestamps salvos no MongoDB e Redis **devem ser armazenados em formato UTC (ISO 8601)**. A conversão para o fuso horário local deve ocorrer apenas no cliente final durante a exibição.

### E. Arquitetura Técnica, Banco de Dados & Collation

1.  **UTF-8 Completo**: Garanta a codificação de caracteres UTF-8 de ponta a ponta (Banco de Dados -> API -> UI) para suportar diversos alfabetos e caracteres especiais.
2.  **Collation em Consultas de Banco**: Diferentes idiomas ordenam caracteres de forma distinta (ex: o caractere `ç` ou letras com acentos no português). Consultas de agregação e listagens de dados no MongoDB devem especificar a Collation adequada:
    *   *Exemplo*:
        ```typescript
        db.collection('workspaces').find().collation({ locale: 'pt', strength: 2 }).sort({ name: 1 })
        ```

### F. Fluxo de Trabalho & Manutenção

1.  **Pseudo-Localização para Testes**: Implementaremos testes visuais utilizando strings expandidas artificialmente (ex: `[!!! Hēllō, {username} !!!]`) para identificar facilmente se elementos de UI quebram devido à expansão ou se existem textos não mapeados.
2.  **Translation Management System (TMS)**: Mapeamento de chaves estruturado para facilitar a futura integração com ferramentas SaaS como Phrase ou Lokalise.

## 🛠️ 4. Cronograma de Implementação (Passo a Passo)

### Fase 1: Fundação & Infraestrutura (Sprint 1)
1.  Instalar dependências fundamentais: `npm install next-intl`.
2.  Configurar a estrutura de mensagens: `messages/pt.json` e `messages/en.json` seguindo os padrões ICU.
3.  Criar `src/i18n.ts` e configurar o middleware híbrido do `next-intl` integrado ao Clerk.
4.  Implementar o script automatizado de verificação de pseudo-localização para testes de layout.

### Fase 2: Reestruturação Física de Rotas Públicas (Sprint 2)
1.  Criar a pasta dinâmica de idioma `src/app/[locale]/`.
2.  Mover rotas públicas do App Router (`ranking`, `v/[id]`, `mentor/[userId]`, `mentorado/[userId]`, `mentorada`, `mentoradx`) para dentro de `src/app/[locale]/`.
3.  Substituir todas as strings das páginas públicas pelo hook de tradução de ICU.

### Fase 3: Localização da Home & Onboarding (Sprint 3)
1.  Adaptar o resolvedor dinâmico do assistente virtual em `HomeContainer.tsx` e `ChatInterface.tsx`.
2.  Substituir a formatação de números e datas nas páginas e popups do fluxo de pagamento utilizando `Intl.NumberFormat` e `Intl.DateTimeFormat`.
3.  Substituir as referências hardcoded de `APP_TAGS` e `APP_ATTRIBUTES` na UI por funções utilitárias que buscam o rótulo traduzido via `useTranslations`.

### Fase 4: Painel Administrativo, Layouts & IA (Sprint 4)
1.  Modificar a barra de navegação global em `AdminNavigation.tsx` e o cabeçalho `AdminTopHeader.tsx`.
2.  Integrar seletor discreto e premium de idiomas no menu de perfil do usuário.
3.  Refatorar os componentes visualmente grandes (`MentoringProfileBoard.tsx` de 145KB, `StoreLayoutGrid.tsx` de 69KB, etc.) substituindo os rótulos de forma segura sem introduzir quebras de layout.
4.  Inserir no gerador back-end (`src/app/api/generate/route.ts` e `src/app/api/generate/tile-stream`) a diretriz `{locale_instruction}` para orientar dinamicamente a geração dos conteúdos no idioma ativo.
